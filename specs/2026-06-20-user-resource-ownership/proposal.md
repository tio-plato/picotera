# 多用户资源归属

在已有的用户与身份模型（`2026-06-18-multi-user`、`2026-06-19-user-management`）之上，引入实际的多用户能力：把密钥、请求、追踪归属到具体用户，并在所有列举、查询、过滤场景中按当前用户做隔离，防止越权。

## 需求

1. **资源用户归属**：给 API Key、请求（request）、追踪（traces）增加用户归属字段。迁移时现有的行全部归入 1 号用户。
2. **网关解析用户**：网关与 unified 路由通过 API Key 解析出当前请求所属用户，放入请求 context。
3. **记录归属**：记录请求、追踪时，将每条请求、每条追踪归属于该 API Key 所属用户。
4. **其它资源不归属**：项目、提供商、模型、端点、脚本、KV、汇率、全局设置等其它资源暂不与用户关联，保持全局共享。
5. **列举隔离**：列出密钥、请求、追踪时，只返回当前用户归属的记录。
6. **越权防护**：获取单个请求/追踪、按条件过滤请求、查看实时进度、中断请求等所有场景都按当前用户限制，防止越权访问他人资源。覆盖 sqlc 中所有相关查询。
7. **概览隔离**：概览（overview）页面的所有指标同步增加用户限制，只统计当前用户的数据。

## 补充确认（规划期间澄清）

- **管理员可见范围**：不设管理员旁路。所有人（含 `is_admin=true`）在列举/查看密钥、请求、追踪、概览时一律仅限本人数据。`is_admin` 仅用于用户管理等其它能力。
- **密钥归属**：API Key 归创建者所有，新建时固定归属当前登录用户，不提供改派字段。现有密钥迁移归 1 号用户。
- **禁用用户的密钥**：网关/unified 路由解析到 API Key 后，若其所属用户已被 `disabled`，拒绝该请求。
