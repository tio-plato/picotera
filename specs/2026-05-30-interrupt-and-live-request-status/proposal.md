# 打断进行中请求 + 进行中请求实时状态

## 需求

### 1. 打断进行中的请求

在控制台上增加按钮，可以打断正在进行中的请求。打断语义按请求行类型区分：

- **打断 upstream 请求**：
  - 若**还没收到 headers**：取消当前这次 upstream 尝试，并**继续走下一个 provider**，行为等同于「这个后端连不上」。
  - 若**已经收到 headers**（正在流式回传）：直接取消，**不再尝试下一个 provider**，行为等同于「后端突然 reset connection」，客户端收到被截断的响应。
- **打断 meta 请求**：直接取消整条请求链路（含当前正在进行的 upstream 尝试），不再尝试任何 provider。

不考虑多实例部署。打断只对运行该请求的本进程生效。

### 2. 进行中请求的实时状态

对尚未完成响应的请求，可以提前查看其实时状态：

- 是否已收到上游响应（headers）。
- 已收到响应的字节数。
- 响应体至今的内容。

这部分数据**只保存在内存里**（用 `sync.Map` 之类的临时结构），查询时从内存读取，**不落库**。请求完成后内存条目即释放。

## 决策约束

- **响应体保留**：内存中**完整保留至今的全部响应体内容**，不截断。仅对进行中的请求保留，完成即释放。
- **UI 位置**：打断按钮和实时状态展示放在**请求详情页的 span 列表**上（meta span 与各 upstream span 均可打断）。
- **刷新方式**：实时状态**不做自动轮询**，由前端在加载详情和用户手动刷新时拉取一次。
- **不新增 DB 字段、不新增 DB status 值**：实时状态完全在内存中，不进入 `request` 表。
- **专用停止原因**：为打断单独定义一个停止原因 `dashboardCancelled`（新增 finish_reason 取值 `7`），与「客户端断开 / 上下文取消」（既有 `Cancelled`）区分开。finish_reason 是普通 int 列、无枚举约束，新增取值不算新增 status 值。
- **map 实现**：进行中请求注册表的并发 map 使用 `github.com/orcaman/concurrent-map/v2`，不手写锁。
