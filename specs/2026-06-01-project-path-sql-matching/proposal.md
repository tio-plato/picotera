# 项目路径匹配改造 + 自动创建项目

## 背景

当前网关请求的项目识别由 `pkg/server/project_extractor.go` + `pkg/server/project_router.go` 完成：
正则从请求体提取候选路径并 JSON 解码，`projectRouter` 把项目表所有 `(project_id, path)` 对加载进内存缓存，按
`strings.HasPrefix(候选路径, 项目路径)`（最长项目路径优先）匹配。任何项目表写入都要手动调用
`Invalidate()`。

## 目标

### 一、改造路径匹配逻辑

- **去掉内存缓存** `projectRouter`，改由数据库用一条 SQL 直接匹配。
- 同时支持 **Windows 与 Unix 两类路径**（反斜线 `\` 与正斜线 `/`）。
- 匹配流程：
  1. 正则提取候选路径，做 JSON 解码得到真实路径（Windows 为单反斜线，Unix 为正斜线）。
  2. 把路径本身加入候选数组。
  3. 从尾部按正斜线或反斜线匹配，每命中一次就把该斜线及其后内容去掉，把得到的祖先路径加入候选数组。
     例如 `/path/to/foo/bar` → `[/path/to/foo/bar, /path/to/foo, /path/to, /path, /]`。
  4. 用一条 SQL 把候选数组交给数据库匹配出符合的项目（命中最长者）。

### 二、全局设置：允许自动创建项目

- 在全局设置中新增「允许自动匹配/创建项目」开关。
- 当开关启用、且本次请求没有匹配到任何项目、且路径长度 > 5 时，自动创建一个项目：
  - 项目名取该路径的最后一个 component（如 `/a/b/foo` → `foo`，`C:\a\foo` → `foo`）。
  - 若已存在同名项目（但其路径未匹配上），则给新项目名加一个随机后缀以避免唯一约束冲突。
  - 在 `project` 表新增一列，标记该项目为自动创建。
  - 新建项目的 `paths` 设为该路径，使后续请求能正常匹配命中。
