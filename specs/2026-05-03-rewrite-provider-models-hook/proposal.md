# 模型列表聚合 hook

## 需求

给脚本功能增加一个 hook 点，用于对模型列表进行处理。在后台渠道那边获取模型列表的时候，创建一个 session 然后激活这个 hook 点。上下文是 provider 的相关字段（包括模型、标注等）、上游原始响应，输入输出是聚合后的模型列表 `[{model: "xx", upstreamModel: "yy"}]` 这样的，是一整个新的 provider models 数组。不直接落库，而是让前端改。

默认聚合规则：原有的模型，先看 `upstreamModel` 字段，如果为空，那就是 `model` 字段，这个统一叫"实际上游模型"；对"实际上游模型"，如果新拉取到的模型列表里面不再存在这个模型了，那么就删掉。再检查新拉取到的模型列表，如果有模型并不存在于老的列表里（以实际上游模型为准判断是否存在），那么就新增这个模型。聚合处理完之后，统一丢给脚本再处理一次，处理完之后的结果放给前端，由前端去 diff，提示用户缺少的模型是否要删除。

处理模型列表时，如果 `(model, upstreamModel)` 完全相同，则视为重复，前后端都做自动去重。

## 决策约束

- **Hook 命名**：`rewriteProviderModels`，与现有 `rewriteModel` / `rewriteRequest` 命名风格一致。
- **聚合基线**：后端从 DB 读 `provider.providerModels` 作为老列表；前端拿到结果后再与自己当前编辑中的 rows 做二次 diff。
- **上游原始响应**：仅在 JS hook ctx 内可见；不出现在 HTTP 响应里。
- **新增条目存储形式**：默认聚合产生的新条目存为 `{model: name}`（省略 `upstreamModel`），与现有约定"空 = 与 model 一致"对齐。
- **去重规则**：以 `(model, upstreamModel)` 完全相同为重复判定；后端聚合前后各做一次，前端在合并 / 保存时再做一次。
