# 管理员扮演用户（Admin Impersonation）

## 原始需求

增加机制使得管理员可以扮演（impersonation）特定用户。在“用户”界面添加按钮，管理员可以点击，此后，前端将该用户的 ID 录入全局状态，并刷新所有的 tanstack/query cache，此后，原本显示用户名的位置将自动变为扮演的用户，但是在用户名右侧显示一个按钮，使得管理员可以还原原本的身份。在使用 API 客户端发送请求时，前端将携带一个特殊的 Header 名为 `X-PicoTera-Impersonation-User-Id`，鉴权中间件若看到此 ID、且已鉴权该用户为管理员，则改为将当前请求识别为管理员要扮演的用户。此机制仅在管理接口生效，而网关不需要此机制。此外，前端发送“测试”请求时，也不需要带上该 Header。

## 补充约定（澄清后确定）

- **真实身份始终来自既有鉴权来源**（single-user-mode / http-header），由服务端可信地解析；扮演 Header 仅是附加提示，只对“真实身份为管理员”的请求生效。
- 扮演对 `GET /api/picotera/me` 同样生效：扮演期间 `me` 返回被扮演用户，因此 `isAdmin`、管理员导航、路由守卫都会切换为被扮演用户的视角；被扮演普通用户时管理员导航与管理页面会自动隐藏/拦截。
- “还原身份”按钮由前端全局状态驱动（不依赖 `isAdmin`），常驻于侧边栏用户名处，因此即使 `isAdmin` 翻转为 `false` 也始终可见。
- 扮演状态仅存于内存（Pinia），不持久化；刷新页面即回到真实身份（安全默认）。
- 开始扮演时跳转到 `/overview`，避免停留在被扮演用户无权访问的管理页面。
- 非管理员携带扮演 Header → 403；Header 非数字 → 400；目标用户不存在 → 404。允许扮演已禁用用户（管理员排查用途）。
- “测试”相关请求（`POST /api/picotera/test/direct` 与网关测试 `postGatewayTest`）使用原生 `fetch`、不经过 openapi-fetch 客户端中间件，天然不携带扮演 Header，无需额外处理。
